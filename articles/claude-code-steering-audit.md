## 公式が言ってる使い分け

ブログの中身をざっくり表にすると、こうです。

| 仕組み | ロードのタイミング | 向いてる用途 |
|--------|------------------|------------|
| CLAUDE.md（ルート） | セッション開始（常時） | 概要・規約・ビルド情報 |
| Rules（paths スコープ） | 該当ファイルに触れたとき | 特定ディレクトリの制約 |
| Skills | 呼び出したとき | 手続き的なワークフロー |
| Subagents | Agent ツール呼び出し | 分離したい脇タスク |
| Hooks | ライフサイクルイベント | 決定的に強制したいこと |

ポイントは一番下。**「絶対やるな」を守らせたいなら、プロンプトに書くんじゃなくて Hooks で止める**、というのが記事の芯でした。

原文の表現だと「A real guardrail needs to be deterministic」。モデルは指示にだいたい従うけど、状況次第で外す。だから本物のガードレールは決定的じゃないとダメ、という話です。

これが自分にいちばん刺さった。

## やらかし①: 「外向き禁止」をプロンプトだけで守らせてた

自分の CLAUDE.md には、こう書いてあった。

> Never directly execute external-facing actions.（外向きの行動を直接実行するな）

でも、これは「お願い」でしかなかった。`git push` を止める仕組みは、どこにも無かった。

つまり、モデルが従ってくれてる間だけ安全、という状態。記事の言う「決定的じゃないガードレール」そのものでした。

### PreToolUse フックで物理的に止めた

`.claude/settings.json` に `PreToolUse` フックを足して、Bash コマンドを実行前にチェックするようにした。

`git push` や PR 作成みたいな外向き・不可逆な操作を見つけたら、**exit code 2 で問答無用にブロック**する。

```bash
# 抜粋: 外向き操作を見つけたら exit 2 で止める
deny='git[[:space:]]+push|gh[[:space:]]+(pr|release|repo)[[:space:]]+create'
if printf '%s' "$cmd" | grep -qiE "$deny"; then
  echo "⛔ 外向き操作です。commit まで止めて、承認を取ってください" >&2
  exit 2
fi
exit 0
```

これで「push していい？」がお願いから強制に変わった。モデルが何を考えてても、push は物理的に通らない。

テストしたら、`git push` は exit 2 でブロック、`git commit` と `gh pr view` は素通り。狙いどおりでした。

## やらかし②: CLAUDE.md がまた 200 行を超えてた

前にも 200 行で痛い目を見たのに、気づいたら 202 行になってた。

中身を見たら、`/ai-ceo:morning` みたいなコマンドの手順が、フォーマット全文ごとベタ書きされてた。30 行超の手続きが、常時ロードされる場所に居座ってた。

記事いわく、手続き的な手順は Skills やコマンド側に置くもの。常時読むファイルに入れるものじゃない。

### 手順をコマンド側に移した

各コマンドの詳しいフローを `.claude/commands/` 側のファイルに移した。CLAUDE.md にはルーティングと原則だけ残した。

結果、202 行 → 147 行。コマンドを実行したときだけ手順が読まれるので、常時のコンテキストも軽くなった。

---

## やらかし③: Skill のファイルの置き方が間違ってた

Skill は `skills/<名前>/SKILL.md` というフォルダー形式で置くのが正解。

なのに自分は `skills/write-blog.md` みたいに、フラットな .md を直置きしてた。これだと Skill として読まれない可能性がある。

6 個中 5 個がこの間違った形だった。気づいてなかった。

### 全部 SKILL.md 形式にそろえた

`write-blog.md` → `write-blog/SKILL.md` のように、フォルダー形式に直した。参照してた箇所のパスも全部追従させた。

地味だけど、これが効いてないと Skill そのものが動かない。基礎の置き方の話でした。

---

## やらかし④: Rules（paths スコープ）を使ってなかった

開発まわりの制約（「新しい依存を入れるなら承認」「スキーマ変更は承認」みたいなやつ）を、全部プロンプト側に混ぜてた。

記事だと、こういう「特定の場所にだけ効く制約」は paths スコープの Rules にするもの。コードに触れたときだけ読まれるので、無駄がない。

### コードに触れたときだけ効くルールにした

`.claude/rules/dev-changes.md` を作って、フロントマターでスコープを切った。

```yaml
---
paths:
  - "products/**"
  - "**/*.ts"
  - "**/*.sh"
---
```

これで、コードやスクリプトを触るときだけ開発ルールが読まれる。普段の会話では出てこない。

---

## ついでに、売り物にも同じ穴があった

ここまで直して気づいたんだけど、配布してる Solo AI Ops Kit にも同じ穴が空いてた。

Skill がフラット形式のまま、外向きを止めるフックも無し。お客さんの環境で Skill が読まれないし、ガードレールも効かない状態。

申し訳ないので、キット側も全部そろえた。SKILL.md 形式に直して、外向きブロックのフックと settings.json を同梱して、`setup.sh` がそれを配るようにした。

導入先で空のプロジェクトに展開するところまでテストして、ちゃんとフックが入ることを確認しました。

---

## 4 つに共通してたこと

全部「仕組みの役割を間違えてた」という話だった。

- 禁止は、プロンプトじゃなくて Hooks で決定的に止める
- 常時ロードの CLAUDE.md には、手続きを置かない
- Skill は、決められた置き方を守る
- 場所限定の制約は、paths スコープの Rules に出す

公式の表に自分を当てはめるだけで、穴が 4 つ見つかった。書いてある側じゃなくて、運用してる側がズレてた、という話でした。
