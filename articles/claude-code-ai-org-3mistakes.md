## この記事のゴール（TL;DR）

一言でいうと、「AIに任せる前に枠を設計する」という話。

読み終わると、AI組織を組み始めたときに踏みやすい3つの落とし穴と、とりあえず動いた対処法が分かります。

- ひとり社長 / 個人開発者で「全部AIに任せられたら」と思ってる人
- Claude Code でエージェントを組んでみたけど、なんかうまく回らない人
- 成功事例より失敗事例を先に読みたい人
- 前提知識: Claude Code を触ったことがあれば十分

## 確認日: 2026-06-21（Claude Code / claude-sonnet-4-6 / Day 20）

## なぜ全部任せようとしたか

社員ゼロの会社を作りたかった。

マーケティング、経理、CS、開発——普通なら何人も雇う部門を、全部 AI に任せたら何が起きるか試したかった。使えるなら使い倒す。失敗するなら記録して公開する。

そう思って始めたのが、今の Homeport です。

CLAUDE.md に 6 部門の役割を書いて、エージェントを 10 体作って、毎朝 GitHub Actions でルーティンを回す。

今日で Day 20。外向きの事故はゼロだった。でも内部では何回もやらかした。

その記録を全部出します。

---

## やらかし①: CLAUDE.md が 200 行を超えた

### 気づいたら 200 行になってた

最初は「詳しく書くほど賢く動く」と思ってた。

部門の役割、ルール、コマンドの書き方、例外処理、禁止事項……気づいたら 200 行を超えてた。

そこからおかしくなった。

エージェントが自分のルールと矛盾した行動を取り始めた。「A をしてはいけない」と「A をしろ」が同じファイルに共存してた。どこで上書きされてるか追えない。

エラーを直してもまた別の矛盾が出てくる。

### 本体を 30 行に削った

CLAUDE.md を 30 行まで絞った。

残したのは「6 部門の一覧」と「外向きの行動は承認制」の 2 本柱だけ。あとは `.claude/agents/` に部門ごとのファイルとして分けた。

動くようになった。

ただ、30 行が最適かどうかは、とりあえずまだわかってない。今のところこれで回ってる、というだけ。

---

## やらかし②: 承認フローがなかった

### 外向きにブレーキがなかった

最初の設計、エージェントが「これが良い」と判断したらそのまま実行できる状態だった。

それで起きたのが、冒頭の話。

ツイートの下書きが本投稿に変わりそうになった。Claude Code が permission に引っかかって止まったけど、設定が少し違えば送信されてた。

実害はゼロだったけど、「危なかった」と実感したのはこのときが初めてだった。

### 外向きを全部 draft 止まりにした

permissions.md を書いた。

「外向きの行動は全部 draft 止まり → 私が承認 → 手動実行」という 3 段構えにした。

ツイート・Zenn 記事・git push——全部このフローを通す。

詳細はこちらに書いた（[AIエージェントが「やりすぎない」設計 — permissions.md 全公開](https://zenn.dev/tomoka/articles/claude-code-permissions-md)）。承認フローの中身が気になる人はこっちを読んでください。

---

## やらかし③: 部門を増やしすぎた

### 9 部門作ったら方向が分からなくなった

最初の 1 週間で部門を 9 つ作った。

成果物の量は増えた。でも方向が分からなくなった。

「マーケ部の記事」「営業部の提案書」「法務部のチェック」が毎日積み上がるのに、どれを先に動かすべきか判断できなくなった。

コストも静かに増えていった。

### 6 部門に絞って月次上限を先に決めた

Dev、Marketing、Sales、Finance、CS、Legal の 6 部門に削った。

月次コストの上限を先に決めた。「この額に収める」という枠が決まると、何を削るかが自動的に決まる。

今はこの 6 部門で動いてる。

---

## 3 つに共通してたこと

全部「任せる前に枠を決めてなかった」という話だった。

CLAUDE.md は行数じゃなくて構造を先に決める。承認フローは「外向きはブレーキを先に設計する」で出発する。部門数は「いくつまで自分が管理できるか」を最初に決める。

枠を先に作れば、AIに丸投げしてもだいたい動く。

逆に、枠なしで丸投げすると——AIは一生懸命に動くけど、方向が分からなくなる。AI が悪いんじゃなくて、設計した側の問題だった。
